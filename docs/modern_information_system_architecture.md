# 北捷品牌數位管家：現代資訊系統架構提案

## 1. 文件目的

本文件從 PM 與系統架構角度，規劃「台北捷運資訊系統 UI/UX 設計指引」未來可擴充的資訊系統架構。

專案目前是一套純靜態網站，主要功能包含 CI 規範查詢、品牌色卡、Logo 使用指引、字體規範、簡報模板、品牌組合預覽、系統版型產生器，以及活動頁與表格範例。現階段優點是部署簡單、使用門檻低、無後端依賴；未來若要支援跨單位維護、權限控管、審核發布、API 串接與正式維運，建議逐步演進為內部品牌資源與介面原型平台。

本文件目標是讓管理者、PM、品牌窗口、資訊維運與開發團隊能用同一張架構藍圖討論後續投資與實作順序。

## 2. 現況盤點

### 2.1 現有系統型態

| 項目 | 現況 |
|------|------|
| 前端 | HTML、CSS、JavaScript |
| 後端 | 無 |
| 資料庫 | 無 |
| 部署 | GitHub Pages 靜態部署 |
| 主要資料 | 寫在 `app.js` 與各頁面 JavaScript 中 |
| 素材 | `assets/`、`files/`、`act_now/images/` |
| 文件 | `README.md`、`docs/` |
| 子模組 | 主站、ACT NOW 活動頁、動態固定欄位表格範例 |

### 2.2 目前核心價值

- 將 PDF、素材包與人工經驗中的 CI 規範轉為可查詢的數位工具。
- 讓使用者能快速複製標準色碼、查看 Logo 使用方式、預覽品牌組合。
- 透過版型產生器降低非技術人員描述系統畫面的門檻。
- 以靜態網站方式提供展示與提案，部署成本低。

### 2.3 目前限制

- 品牌資料、版型資料與範例資料寫在 JavaScript 中，維護需要開發者介入。
- 缺少權限控管與審核流程，無法區分草稿、審核中、已發布版本。
- 素材缺少 metadata，例如來源、版本、審核人、適用情境與停用狀態。
- 其他內部系統無法透過 API 取得最新 CI 色碼、Logo 與模板。
- 缺少正式的資料備份、監控、稽核紀錄與環境分層。

## 3. 目標定位

建議將本專案定位為「北捷內部品牌與介面原型平台」。

平台不只是靜態展示頁，而是提供以下能力：

- 品牌規範查詢：統一查詢色卡、Logo、字體、文件與模板。
- 素材資產管理：集中保存素材，並保留版本、來源與審核狀態。
- 原型產生工具：支援業務單位快速產生符合 CI 的系統版型。
- 後台維護流程：讓授權人員可新增、修改、送審與發布資料。
- API 服務：讓其他內部系統可使用最新品牌資料與設計 token。
- 使用分析：掌握常用素材、常用模板與跨單位需求趨勢。

## 4. 建議整體架構

### 4.1 架構分層

建議採用分層式架構，避免前端、資料、審核與部署邏輯混在同一份靜態程式碼中。

| 層級 | 角色 | 建議技術 |
|------|------|----------|
| 使用者介面層 | 前台網站、後台管理、版型產生器 | React、Vue，或先以原生 JS 模組化過渡 |
| API 服務層 | 提供品牌資料、素材、模板、審核與統計 API | Node.js/NestJS、FastAPI 或 ASP.NET Core |
| 業務服務層 | 權限、審核、版本、模板產生、素材處理 | Service layer、背景工作佇列 |
| 資料層 | 保存結構化資料與素材 metadata | PostgreSQL |
| 檔案儲存層 | 保存圖片、PDF、PPTX、壓縮包 | MinIO、S3 相容儲存或既有內部檔案服務 |
| 快取層 | 熱門資料、session、背景任務狀態 | Redis |
| 部署維運層 | 容器化、反向代理、環境設定 | Docker Compose、Nginx/Traefik |
| 監控稽核層 | log、錯誤追蹤、異動紀錄 | OpenTelemetry、Prometheus、Grafana 或既有監控平台 |

### 4.2 邏輯架構圖

```text
使用者 / 內部系統
        |
        v
反向代理 / HTTPS / 路由
        |
        +--------------------+
        |                    |
        v                    v
前台網站 / 原型工具        後台管理系統
        |                    |
        +---------+----------+
                  |
                  v
              API 服務層
                  |
        +---------+----------+--------------+
        |                    |              |
        v                    v              v
     PostgreSQL            Redis        物件儲存
   結構化資料             快取/任務       素材檔案
```

## 5. 核心子系統設計

### 5.1 前台網站

前台維持目前使用者熟悉的功能，但資料來源改由 API 或靜態 JSON 設定檔提供。

建議功能：

- 品牌色卡查詢、搜尋、分類、排序與複製。
- Logo 使用規範、背景示意、下載與使用限制說明。
- 字體與簡報模板展示。
- 品牌組合預覽。
- 系統版型產生器。
- 素材下載中心。
- 文件與版本公告。

短期可保留目前靜態頁，先將資料拆出；中期再導入前端框架與元件化。

### 5.2 後台管理系統

後台是本專案從展示工具升級為正式平台的關鍵。

建議功能：

- 色卡資料管理：色名、HEX、分類、用途、排序、狀態。
- Logo 管理：版本、檔案、適用背景、禁止事項、下載權限。
- 模板管理：簡報模板、系統版型、業務場景、預覽圖。
- 素材管理：上傳、分類、版本、來源、授權、審核狀態。
- 文件管理：CI 文件、操作說明、異動公告。
- 審核流程：草稿、送審、退回、發布、停用。
- 稽核紀錄：誰在何時修改、審核、發布或停用資料。

### 5.3 API 服務

API 應採版本化設計，避免未來調整破壞既有內部系統。

第一版建議使用 REST API：

| API | 用途 |
|-----|------|
| `GET /api/v1/brand/colors` | 取得品牌色卡 |
| `GET /api/v1/brand/logos` | 取得 Logo 清單與下載資訊 |
| `GET /api/v1/brand/fonts` | 取得字體規範 |
| `GET /api/v1/templates/presentations` | 取得簡報模板 |
| `GET /api/v1/layout-seeds` | 取得版型產生器業務場景 |
| `POST /api/v1/layouts/generate` | 依參數產生版型 HTML 或 JSON |
| `GET /api/v1/assets` | 查詢素材資產 |
| `POST /api/v1/admin/assets` | 上傳素材 |
| `POST /api/v1/admin/reviews/{id}/approve` | 審核發布 |
| `GET /api/v1/audit-logs` | 查詢稽核紀錄 |

API 回應應包含版本、發布狀態與最後更新時間，讓前端與串接系統能判斷資料新鮮度。

### 5.4 素材與檔案服務

素材不建議直接存入資料庫。資料庫保存 metadata，實體檔案保存於物件儲存或內部檔案服務。

metadata 建議包含：

- 素材名稱
- 類型，例如 Logo、簡報、PDF、活動圖、壓縮包
- 檔案格式與大小
- 版本
- 來源
- 授權或使用限制
- 審核狀態
- 建立者、審核者、發布日期
- 下載 URL

### 5.5 權限與角色

建議整合內部 SSO、AD 或 LDAP。若第一階段沒有可用整合，可先用本地帳號實作。

| 角色 | 權限 |
|------|------|
| 檢視者 | 查詢規範、下載允許素材、使用版型產生器 |
| 編輯者 | 建立與修改草稿資料 |
| 審核者 | 退回、核准、發布或停用資料 |
| 系統管理者 | 管理帳號、角色、系統設定、稽核查詢 |

重要原則：

- 前台只顯示已發布資料。
- 後台修改不應直接影響正式資料。
- 所有發布行為必須留下稽核紀錄。
- 素材下載可依角色或單位控管。

## 6. Docker 環境設計

### 6.1 初期 Docker Compose 組成

```text
frontend
  - 提供前台與後台前端靜態檔

api
  - 提供 REST API、權限、審核、版本管理

postgres
  - 保存品牌資料、素材 metadata、審核紀錄

redis
  - 快取熱門查詢、session、背景任務狀態

minio
  - 保存圖片、PDF、PPTX、壓縮包等素材

worker
  - 處理縮圖、匯入匯出、統計彙整、檔案掃描

reverse-proxy
  - HTTPS、路由、安全標頭、壓縮與快取策略
```

### 6.2 環境分層

建議至少分為三個環境：

| 環境 | 用途 |
|------|------|
| local | 開發者本機，使用 Docker Compose 啟動 |
| staging | PM、品牌窗口、測試人員驗收 |
| production | 正式服務，僅發布已審核內容 |

`.env` 應管理資料庫連線、儲存服務、SSO 設定、API URL、檔案大小限制與 log level。正式密碼不得提交到 Git。

## 7. CI/CD 與維運流程

目前專案已有 GitHub Pages 部署流程。未來導入 Docker 與 API 後，建議升級為以下流程：

1. Pull Request 建立後，自動執行 lint、unit test、build。
2. 合併到主分支後，建立前端與 API Docker image。
3. 自動部署到 staging。
4. PM 與品牌窗口完成驗收。
5. 經人工核准後部署 production。
6. 發布完成後建立版本紀錄與 changelog。
7. 若發生問題，可回滾到前一版 image 與資料版本。

建議檢查項目：

- 前端建置成功。
- API schema 測試通過。
- 資料庫 migration 可正常執行與回滾。
- Docker image 沒有高風險弱點。
- 關鍵頁面可正常開啟。
- 後台權限與審核流程可正常運作。

## 8. 資料治理與版本策略

品牌資料具有正式性，應比一般內容更重視來源、審核與版本。

建議資料狀態：

| 狀態 | 說明 |
|------|------|
| draft | 編輯中，不提供前台使用 |
| review | 已送審，等待品牌或 PM 確認 |
| published | 已發布，前台與 API 可使用 |
| archived | 已停用，保留歷史紀錄 |

建議版本策略：

- 色卡、Logo、模板與 CI 文件都應保存版本。
- 新版本發布後，舊版本不立即刪除，改為封存。
- API 回應提供 `version` 與 `updatedAt`。
- 重大變更需建立公告，避免使用者沿用舊素材。

## 9. 資安與合規考量

本平台主要處理品牌資料與素材，但仍需符合內部資訊安全基本要求。

建議控管：

- 後台必須登入。
- 管理 API 必須檢查角色權限。
- 上傳檔案限制副檔名、大小與 MIME type。
- 下載紀錄需可追蹤。
- 檔案不得使用使用者原始檔名直接作為儲存 key。
- API 必須有 rate limit 與錯誤回應標準。
- 正式環境必須使用 HTTPS。
- log 不記錄密碼、token 或敏感資訊。

## 10. 分期推動路線圖

### 第一階段：穩定現有靜態工具，0 到 3 個月

目標是降低維護成本，不急著導入完整後端。

建議工作：

- 將色卡、Logo、字體、模板與版型場景從 JavaScript 拆為 JSON。
- 整理素材命名、來源與文件索引。
- 建立發布檢查表與版本紀錄。
- 保留 GitHub Pages 靜態部署。
- 補上基本前端檢查與文件連結檢查。

### 第二階段：建立 API 與 Docker 開發環境，3 到 6 個月

目標是把系統從靜態工具升級為可服務化的平台基礎。

建議工作：

- 建立 API 專案與資料庫 schema。
- 建立 Docker Compose，包含 frontend、api、postgres、redis、minio。
- 前台改由 API 或靜態 JSON 讀取資料。
- 建立 staging 環境。
- 加入 API 文件，例如 OpenAPI/Swagger。

### 第三階段：建立內部正式平台，6 到 12 個月

目標是支援跨單位正式使用。

建議工作：

- 導入登入與角色權限。
- 建立後台管理系統。
- 建立審核發布流程。
- 建立素材上傳與版本管理。
- 建立稽核紀錄與使用分析。
- 建立 production 部署與備份還原流程。

### 第四階段：平台化與跨系統服務，12 個月以上

目標是讓平台成為內部設計系統與品牌 API 的來源。

建議工作：

- 提供設計 token API。
- 提供內部系統串接範例。
- 建立範本中心與元件規範。
- 建立常用業務系統原型模板。
- 評估 AI 輔助產生介面原型。
- 導入更完整的監控、容量規劃與服務等級管理。

## 11. 成功指標

| 指標 | 衡量方式 |
|------|----------|
| CI 查詢效率 | 使用者取得色碼、Logo 或模板所需時間 |
| 品牌一致性 | 錯誤色碼、錯誤 Logo 或過期模板使用次數 |
| 服務採用率 | 每月使用人數、使用單位數、API 呼叫數 |
| 原型產出效率 | 從需求討論到初版畫面產出的時間 |
| 維護效率 | 更新色卡、Logo、模板所需時間 |
| 審核品質 | 發布後需回修或撤回的比例 |
| 平台穩定度 | 正式服務可用率、錯誤率、平均回應時間 |

## 12. 建議優先順序

若資源有限，建議依照以下順序推動：

1. 資料拆分：先將寫死在 JavaScript 的資料移出，降低維護成本。
2. 素材治理：建立素材清單、版本、來源與審核狀態。
3. API 化：提供色卡、Logo、模板與版型資料 API。
4. Docker 化：建立一致的本機與測試環境。
5. 後台與審核：讓非工程角色可維護正式資料。
6. 權限與稽核：支援正式內部平台要求。
7. 跨系統串接：讓其他內部系統使用品牌資料與設計 token。

## 13. 結論

本專案已具備成為內部品牌資源平台的雛形。現階段不建議一次重寫成大型系統，而應採取「先資料化、再服務化、最後平台化」的演進策略。

短期重點是保留靜態網站的低門檻優勢，同時把資料從程式碼中拆出；中期導入 API、Docker 與資料庫，建立正式服務基礎；長期再加入權限、後台、審核、稽核、API 串接與 AI 輔助原型能力。

透過這樣的架構，本工具可從一次性的提案展示，逐步成為可維護、可審核、可串接、可擴充的現代資訊系統。
