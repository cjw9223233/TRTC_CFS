# 維護手冊與未來規模化發展策略

## 一、文件目的

本文件從 PM 與系統架構師角度，說明「台北捷運資訊系統 UI/UX 設計指引」專案的維護方式、版本管理、品質檢查，以及未來系統規模化後可延伸的發展方向。

本專案目前是一套純靜態網頁工具，主要提供 CI 規範查詢、品牌色卡、Logo 使用指引、字體規範、簡報模板、品牌組合預覽與系統版型產生功能。未來若使用人數、資料量與跨單位需求增加，可逐步演進為內部品牌資源平台。

## 二、系統現況與定位

### 2.1 目前定位

本系統定位為「北捷品牌數位管家」的展示與實用工具，讓非技術同仁也能快速查詢品牌規範、複製色碼、預覽品牌組合，並產出符合 CI 風格的系統介面原型。

### 2.2 目前技術型態

| 項目 | 現況 |
|------|------|
| 前端型態 | 純 HTML / CSS / JavaScript |
| 啟動方式 | 直接以瀏覽器開啟 `index.html` |
| 後端服務 | 無 |
| 資料庫 | 無 |
| 主要資料來源 | `app.js` 內的色卡、字體、Logo、模板與版型設定 |
| 文件管理 | `docs/` 目錄 |
| 素材管理 | `assets/` 與 `files/` 目錄 |

### 2.3 核心使用情境

- 查詢台北捷運品牌色碼與 CI 規範。
- 複製 HEX 色碼，套用於簡報、文件或網頁。
- 查看 Logo 在不同背景下的使用建議。
- 預覽品牌色、Logo、字體與簡報模板組合。
- 產生業務系統介面原型，協助需求討論。
- 作為 Vibe Coding / AI 輔助開發成果展示。

## 三、維護角色與責任分工

| 角色 | 主要責任 |
|------|----------|
| PM / 產品負責人 | 管理需求優先順序、確認使用情境、安排版本發布、彙整使用者回饋 |
| 品牌 / 設計窗口 | 確認 CI 規範、色碼、Logo、字體與簡報模板是否正確 |
| 前端維護者 | 維護 `index.html`、`style.css`、`app.js`，處理互動功能與版面問題 |
| 文件維護者 | 維護 `README.md`、`docs/` 文件與操作說明 |
| 審核窗口 | 發布前確認品牌、內容、法務或資安相關風險 |

## 四、日常維護作業

### 4.1 品牌色卡維護

當官方 CI 色碼調整或新增時，需同步更新系統內色卡資料。

建議流程：

1. 由品牌窗口提供正式色碼來源。
2. 前端維護者更新 `app.js` 中的色卡資料。
3. 檢查色名、HEX、分類、排序與複製功能。
4. 文件維護者更新 README 或相關操作文件。
5. 發布前由品牌窗口再次確認。

### 4.2 Logo 與素材維護

Logo 圖檔、壓縮包與品牌素材建議放置於 `files/` 或 `assets/`。

維護原則：

- 檔名應清楚描述用途與版本。
- 若 Logo 有版本差異，需保留可辨識的命名。
- 避免刪除舊素材，除非已確認沒有頁面或文件引用。
- 若替換正式 Logo，應記錄來源、日期與審核人。

### 4.3 文件維護

提案、操作說明、維護手冊與決策紀錄建議放置於 `docs/`。

維護原則：

- 新增文件後，應同步更新 `README.md` 的文件索引。
- 文件應使用繁體中文與 UTF-8 編碼。
- 文件標題應清楚表達用途，避免只用日期或模糊名稱。
- 若文件涉及正式對外內容，需經 PM 與品牌窗口確認。

### 4.4 版型與功能維護

目前系統的版型產生器與品牌產生器主要由前端 JavaScript 控制。若新增業務場景、版面結構或輸出模板，需確認：

- 是否符合 CI 色彩與字體規範。
- 是否能在桌面與手機版正常顯示。
- 產出的 HTML 是否可讀、可複製、可作為原型使用。
- 不同瀏覽器的互動行為是否一致。

## 五、版本管理與發布流程

### 5.1 Git 管理建議

建議採用簡單、可追蹤的版本管理方式：

1. 每次修改建立獨立分支，例如 `feature/update-brand-colors`。
2. 修改完成後提出 Pull Request 或 Merge Request。
3. 至少由 PM 或品牌窗口進行內容確認。
4. 發布後建立版本標籤，例如 `v1.1.0`。
5. 在文件中記錄重要變更。

### 5.2 版本命名建議

| 類型 | 範例 | 說明 |
|------|------|------|
| 修補版 | `v1.0.1` | 修正錯字、連結、樣式小問題 |
| 功能版 | `v1.1.0` | 新增色卡、模板、範例或互動功能 |
| 大版本 | `v2.0.0` | 架構調整，例如導入資料檔、後台或部署流程 |

### 5.3 發布前基本流程

1. 確認檔案可正常開啟。
2. 確認主要功能可操作。
3. 確認文件連結正確。
4. 確認品牌內容已審核。
5. 確認沒有不應公開的內部資料。
6. 建立版本紀錄並通知使用者。

## 六、品質檢查清單

### 6.1 功能檢查

- `index.html` 可正常開啟。
- 品牌色卡可正常顯示。
- 色碼複製功能可使用。
- 搜尋、分類與排序功能可使用。
- Logo、字體、簡報模板區塊內容正確。
- 品牌產生器可即時更新預覽。
- 版型產生器可產生並複製 HTML。
- `act_now/index.html` 應用範例可正常瀏覽。

### 6.2 內容檢查

- 色碼與官方來源一致。
- Logo 使用說明符合最新 CI 規範。
- 文件沒有錯字、亂碼或過期資訊。
- Markdown 連結可正確開啟。
- 對外展示內容沒有未授權素材或敏感資訊。

### 6.3 相容性檢查

- Chrome 可正常使用。
- Microsoft Edge 可正常使用。
- 桌面版版面不重疊、不跑版。
- 手機版可閱讀主要內容。
- 離線開啟時主要內容仍可使用。

## 七、風險與控管策略

| 風險 | 影響 | 控管策略 |
|------|------|----------|
| CI 規範過期 | 使用者套用錯誤品牌資訊 | 建立品牌窗口審核與版本註記 |
| 資料寫死在 `app.js` | 更新成本上升 | 中期拆分為 JSON 設定檔 |
| 文件未同步更新 | 新手使用者無法正確操作 | 發布前檢查 README 與 docs |
| 素材來源不明 | 對外使用可能產生授權疑慮 | 素材需標記來源與審核狀態 |
| 功能擴充後難維護 | 前端檔案過大、邏輯耦合 | 導入模組化架構與自動化測試 |
| 使用者自行修改版本 | 各單位版本不一致 | 建立正式發布頁與版本公告 |

## 八、未來規模化發展方向

### 8.1 短期：整理靜態工具，降低維護成本

短期目標是讓現有工具更容易維護，並確保新同仁能快速接手。

建議方向：

- 將色卡、Logo、字體、模板設定逐步從 `app.js` 拆成獨立資料檔。
- 補齊操作教學、維護手冊與異動紀錄。
- 建立固定檢查清單，避免發布時漏測。
- 將提案文件、操作文件與正式素材分區管理。

### 8.2 中期：導入平台化管理能力

當使用者增加、資料更新頻率提高後，可導入管理後台與自動化流程。

建議方向：

- 建立品牌資料管理後台，讓授權人員維護色卡、Logo 與模板。
- 導入使用者權限，區分檢視者、編輯者、審核者與管理者。
- 建立審核流程，品牌資料需審核後才可發布。
- 導入 CI/CD，自動檢查連結、格式與基本前端錯誤。
- 將工具部署到內部網站，建立唯一正式入口。

### 8.3 長期：發展為內部品牌與介面生成平台

長期可將本工具擴展為跨單位使用的品牌設計與系統原型平台。

建議方向：

- 建立品牌素材庫，集中管理 Logo、圖示、模板、簡報與設計元件。
- 提供 API，讓其他內部系統可直接取得最新 CI 色碼與樣式設定。
- 建立範本中心，提供常用頁面、報表、Dashboard 與表單版型。
- 導入 AI 輔助介面生成，讓使用者以自然語言描述需求後產生初版畫面。
- 累積使用數據，分析最常用素材、功能與業務場景。
- 發展設計系統，形成可重複使用的元件、樣式與規範。

## 九、階段性推動路線圖

| 階段 | 時程建議 | 目標 | 主要成果 |
|------|----------|------|----------|
| 第一階段 | 0-3 個月 | 穩定現有靜態工具 | README、維護手冊、文件索引、發布檢查清單 |
| 第二階段 | 3-6 個月 | 降低資料維護成本 | 拆分設定檔、整理素材庫、建立版本紀錄 |
| 第三階段 | 6-12 個月 | 建立內部正式服務 | 內部部署、權限控管、審核流程、CI/CD |
| 第四階段 | 12 個月以上 | 發展平台化能力 | API、範本中心、AI 介面生成、設計系統 |

## 十、建議成功指標

| 指標 | 衡量方式 |
|------|----------|
| CI 查詢效率 | 使用者取得色碼或 Logo 規範所需時間 |
| 品牌一致性 | 文件、簡報與系統畫面中錯誤色碼或錯誤 Logo 使用次數 |
| 使用者採用率 | 每月使用人數、使用單位數、下載或複製次數 |
| 原型產出效率 | 從需求討論到產出初版畫面的時間 |
| 維護效率 | 更新色卡、文件或模板所需時間 |
| 審核品質 | 發布後回報錯誤或需回修的比例 |

## 十一、結論

本專案目前已具備可展示、可查詢、可複製、可產生原型的基本能力，適合作為北捷 CI 規範數位化的第一步。短期應優先確保內容正確、文件完整與維護流程清楚；中期可導入資料拆分、權限與審核流程；長期則可擴展為內部品牌資源平台與介面生成平台。

以 PM 角度來看，這套工具的價值不只是展示 AI 輔助開發成果，而是把原本分散在 PDF、素材包與個人經驗中的品牌知識，轉換成可被查詢、可被套用、可被維護的數位服務。以系統架構角度來看，現階段應保留靜態工具的低門檻優勢，同時預留未來資料化、平台化與服務化的演進路徑。
